# Day3

## Call Stacks
The primary use of a callstack is for funtions in a program to 
1. spill fast-call register values
2. store local variables
3. store return addresses
4. pass additional arguments

Note: always compile in debug mode. This will prevent the compiler from optimizing the final code. 

 Argument slot    Integer / pointer register    Floating-point register
  ━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━  ━━━━━━━━━━━━━━━━━━━━━━━━━

   3rd              R8                            XMM2
  ───────────────  ────────────────────────────  ─────────────────────────
   4th              R9                            XMM3

Shadow region -> reserve an 8 byte chunk on the stack for each of these registers. It's just a guaranteed stack slot for the 4 arguments. If the function wants to reuse RCX, RDX, R8, R9, it can do so without checking where it is or allocating first.  The high level reasoning is that it's just to reduce the amount of operations since the values are reused quite a bit. 

[rsp+20h] <- for r9

[rsp+18h] <- for r8

[rsp+10h] <- for rdx

[rsp+08h] <- for rcx


Stack frame → a region of the stack relevant to the currently executing function. 

When a new function is called, a new frame is setup to store that function’s data such as local variables and the return address of the caller. The return address is always the first piece of data in the stack frame, which is pushed to the stack when a call instruction is executed. 

This is only applicable after the call has happened. Not when preparing for main. You can see later down below.
```
[rsp+30h] <- arg6
[rsp+28h] <- arg5
[rsp+20h] <- for r9
[rsp+18h] <- for r8
[rsp+10h] <- for rdx
[rsp+08h] <- for rcx
[rsp] <- return address
```
