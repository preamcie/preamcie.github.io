# Beacon concepts

This is what a COFF is. 

```
┌─────────────────────────────┐
│      COFF Object File        │
├─────────────────────────────┤
│ File Header                  │ ← Machine type, section count
├─────────────────────────────┤
│ Section 1: .text (code)      │ ← Your actual instructions
│   - Function A               │
│   - Function B               │
├─────────────────────────────┤
│ Section 2: .data (data)      │ ← Global variables, strings
├─────────────────────────────┤
│ Section 3: .rdata (readonly) │ ← Constants, import names
├─────────────────────────────┤
│ Symbol Table                 │ ← What functions are here?
│   - Function A (internal)    │
│   - Function B (exported)    │
│   - _GetProcAddress (needed) │
├─────────────────────────────┤
│ Relocation Information       │ ← "Fix this address when loaded"
└─────────────────────────────┘
```

## Storyline

In a real assessment, you need to get initial access, and you want to get a shell first from anything, then inject into explorer.exe or svchost.exe. However, the lab is just about seeing if the loader can bypass EDR alert, so we compile it into an .exe and run it. 

Basically you have a .dll payload that Cobalt Strike produces. We don't use .exe, because we want to use .dll to inject in other processes like svchost.exe or explorer.exe. 

You want to append the PIC to it at the front, not in the .text section. 

Usually, to run a .dll, you have to use LoadLibrary(). However, Windows will know about it, it gets registered in the process's module list, the EDR gets notified. 

That's why we redo a loader - reimplement ParseDLL, LoadDLL, ProcessImports without telling anyone. 

We need to 
1. Parse the DLL's PE headers
2. Allocate memory
3. Copy each section to the correct location
4. Resolve the DLL's imports (fill in its IAT)
5. Set section permissions
6. Call the entry point. 

Cobalt Strike always produces Beacon as a DLL. The output format you choose (EXE, DLL, shellcode, etc.) only changes the delivery wrapper — what's inside is always the loader PIC + Beacon DLL.

```
[ Loader (PIC) ][ Beacon (DLL) ]
```

The EXE is just a delivery container so you can double-click and run it. When the EXE runs, it executes the loader PIC, which then loads the Beacon DLL in memory.

## How PicoLoad works

PICO is crystal palace's own lightweight PIC format for smaller payloads. If something is compiled as a PICO instead of a DLL, the loader should use PicoLoad instead of the full ParseDLL -> LoadDLL -> ProcessImports chain, as there are no PE headers. 


## Crystal Palace high level overview

Loader side

loader.c       — go() function, the main entry point
hooks.c        — _VirtualAlloc, _VirtualProtect, _LoadLibraryA (loader's hooks)
spoof.c        — Draugr spoofing code
draugr.asm     — assembly stub (builds fake stack frames, saves the original registers)
services.c     — resolve functions for DFR
loader.spec    — combines everything, uses attach to hook loader's API calls

These hooks only affect API calls the loader makes while loading Beacon.

PICO side

pico.c         — _GetProcAddress and setup_hooks
hooks.c        — _InternetConnectA, _OpenProcess, etc. (Beacon's hooks)
spoof.c        — same Draugr spoofing code (merged again)
draugr.asm     — same assembly stub (linked again)
pico.spec      — combines everything, uses addhook to register which APIs to intercept
